# 🔋 Motorola IMPRES Battery Reader / Programmer (ESP32)

Пристрій на **ESP32** для **зчитування, аналізу, редагування, відновлення та
програмування** чипів акумуляторів Motorola IMPRES — через веб-інтерфейс
Wi-Fi, USB-клієнт (браузер **або** нативний `.exe`) і OLED/LCD-дисплей з
кнопковим меню.

Інтерфейс українською, заставка з емблемою **Національної Гвардії України**.

Всередині акумулятора IMPRES два 1-Wire чипи:
- **DS2433** — EEPROM 512 байт: модель, крива розряду, історія ємності,
  лічильники, блок автентифікації.
- **DS2438** — монітор батареї: напруга, струм, температура, лічильники
  заряду/розряду (ICA/CCA/DCA), калібрування.

> ⚠️ **Family codes для ЦЬОГО заліза — `DS2433_ID = 0xA3`, `DS2438_ID = 0xA6`**
> (перевірено осцилографом, у `battery_reader.h`). Це **не** стандартні
> 0x23/0x26 з даташита. Заміна ламає зчитування — **не чіпати**.

- ♻️ **Скидання** лічильників/зносу для рекалібрування на IMPRES-ЗП.
- 🛠 **Ремонт цілісності** (контрольні суми + дзеркало калібрування).
- 🧹 **Очистка** (стерти все, крім ідентичності/калібрування).
- 🔥 **Повне стирання DS2433** (крайній випадок).
- 🪪 **Ручний запис моделі** (part number) з перевіркою довжини/символів.
- ✏️ **Редактор байтів** із **лінійкою адрес і ASCII-колонкою** (як у HxD),
  зміна **ємності в мА·год** і **зносу %**.
- 🔁 **Перезавантаження ESP32** з веб/USB/меню дисплея.
- 🔐 Пароль **обов'язковий у веб-API**, **опційний по USB** (фізичний доступ).
- 🛡 **Модельно-залежна перевірка справжності** (формати 2017 та 2014/4409A).
- 🎨 Підтримка **8 моделей дисплеїв** (I2C/SPI, у т.ч. **кольорові TFT
  ST7789** 240×240 / 240×280 з кольоровим меню й заставкою) зміною однієї опції.
- 💡 Неблокуюча **індикація світлодіодами**.
- 🔬 Задокументований **реверс формату IMPRES** (контрольні суми, дзеркало,
  прив'язка до моделі, а не до серійника).

## 1. Обладнання та підключення

**Плата:** будь-яка ESP32. **Бібліотеки** (Arduino Library Manager):
`U8g2`, `OneWire`, `ArduinoJson`.

| Сигнал | GPIO | Константа | Примітка |
|---|---|---|---|
| 1-Wire дані (DS2433/DS2438) | **13** | `DS_PIN` | до контакту даних АКБ |
| Керування підтяжкою 1-Wire | **12** | `PULLUP_PIN` | сильний pull-up на час запису |
| LED зелений | **14** | `LED_GREEN_PIN` | через ~330 Ом на GND |
| LED червоний | **27** | `LED_RED_PIN` | через ~330 Ом на GND |
| Кнопка «Вперед» | **25** | `MENU_BTN_PIN` | між GPIO і GND (LOW) |
| Кнопка «Назад» | **26** | `MENU_BTN2_PIN` | між GPIO і GND (LOW) |

Між лінією даних (GPIO13) і 3.3V — підтяжка **2.2…4.7 кОм**. Усі піни
змінюються у `settings.h`.

## 2. Вибір дисплея

У `settings.h`, **КРОК 1** — розкоментуйте **рівно один** рядок:

```c
#define DISPLAY_SSD1306_I2C        // 0.96" 128x64 I2C (GME12864) — за замовч.
// #define DISPLAY_SH1106_I2C      // 1.3"  128x64 I2C
// #define DISPLAY_SH1107_128_I2C  // GME128128-02, 128x128 I2C (SH1107)
// #define DISPLAY_SSD1327_128_I2C // GME128128-02, 128x128 I2C (SSD1327)
// #define DISPLAY_ST7567_SPI      // Open-Smart 1.8" 128x64 SPI
// #define DISPLAY_PCD8544_SPI     // Nokia 5110 (PCD8544) 84x48 SPI
// --- КОЛЬОРОВІ TFT на ST7789 (SPI) — кольорове меню + кольорова заставка ---
// #define DISPLAY_ST7789_SPI
//   #define DISPLAY_ST7789_240X240 // ST7789VW 240x240
//   #define DISPLAY_ST7789_240X280 // ST7789V3 240x280
```

Роздільність, розкладка меню і шрифти підставляються автоматично.

- **I2C** (КРОК 2a): `SDA=21`, `SCL=22`, адреса `0x3C`.
  `#define DISPLAY_HW_I2C` — апаратний I2C. Якщо він не відповідає — прошивка
  **сама переходить** на програмний I2C (у Serial: `HW I2C OK` / `SW I2C
  fallback`). Часта причина відмови — слабкі підтяжки: зовнішні **4.7 кОм**.
  `DISPLAY_I2C_KHZ 0` = авто (SSD1327 тримає лише 100 кГц).
- **SPI** (КРОК 2b): `SCK=18`, `MOSI=23`, `CS=5`, `DC=17`, `RST=16`.
  ST7567: зсув `DISPLAY_ST7567_XOFF` (0…4); контраст `DISPLAY_CONTRAST`.

### 🎨 Кольорові TFT (ST7789VW 240×240 / ST7789V3 240×280)

Розкоментуйте `DISPLAY_ST7789_SPI` **і рівно один** розмір. Меню, заставка й
індикатори — **кольорові** (заряд: зелений/жовтий/червоний; небезпечні дії —
червоним; заставка НГУ синьо-жовта). Уся кирилиця збережена.

- **Потрібні бібліотеки** (Менеджер бібліотек Arduino): `Adafruit GFX Library`,
  `Adafruit ST7735 and ST7789 Library`, `Adafruit BusIO`, `U8g2_for_Adafruit_GFX`.
- **Піни SPI:** `SCK=18`, `MOSI(SDA)=23`, `CS=5`, `DC(RS)=17`, `RST=16`
  (спільні з КРОК 2b). Підсвітка — необов'язково: `#define DISPLAY_BLK_PIN 4`.
- **Орієнтація/інверсія:** `DISPLAY_ST7789_ROT 0…3`, `DISPLAY_ST7789_INVERT`.
  Якщо на 240×280 картинка зсунута по вертикалі — оновіть бібліотеку
  Adafruit ST7789 (свіжі версії враховують зсув 240×280) або змініть орієнтацію.
- **Заокруглені кути** (ST7789V3 1.69" 240×280 зазвичай скруглений): увімкніть
  `#define DISPLAY_ST7789_ROUND` — інтерфейс додасть безпечний відступ від кутів,
  щоб заголовок, індикатор сторінки, рядок статусу й hex не обрізались дугою.
  Радіус за потреби — `#define DISPLAY_ST7789_CORNER 22`.
- Реалізація — окремий файл `display_color.h`; монохромний u8g2-шлях не
  зачіпається (обидва варіанти взаємовиключні).
- **SSD1327**: якщо екран пустий/спотворений — панелі бувають з різним
  контролером-remap: у `settings.h` спробуйте `SSD1327_WS` / `SSD1327_EA` /
  `SSD1327_ZJY` (за замовч. MIDAS).

Активний дисплей і частоту шини видно в Serial (115200) при старті.

## 4. Веб-інтерфейс

Підключіться до Wi-Fi **`BatteryReader_moto`** (пароль `12345678`). **Captive-
portal** сам запропонує відкрити сторінку; інакше — `http://192.168.4.1`.
Пароль для запису/операцій — **`admin123`** (`ADMIN_PASSWORD`), **обов'язковий**
для всіх операцій запису.

- **Огляд** — заряд %, залишок мА·год, напруга, температура, модель, знос,
  цикли, справжність, цілісність.
- **Дані** — модель/серійний, DS2438 (струм, ICA/CCA/DCA в мА·год і циклах),
  повні hex-дампи.
- **Прошивка** — згруповано за рівнем ризику: **Резервна копія** →
  **Обслуговування** (безпечно для ідентичності) → **Небезпечна зона**
  (незворотні операції). Тут же **ручний запис моделі**, **🆕 Новий акумулятор**,
  **редактор байтів** із лінійкою адрес і ASCII-колонкою (як у HxD), а також
  **🔁 Перезавантаження**. Див. [розділ 8](#8-операції-з-акб).

> 🪪 **Серійний номер** — це апаратний лазерний ROM чипа DS2438: він унікальний,
> незмінний і **фізично не перезаписується**. Записується лише **модель**
> (part number). Для власного інвентарного ID використайте редактор байтів.

## 5. USB-клієнт

Той самий функціонал по кабелю, працює **паралельно** з Wi-Fi. Три варіанти —
оберіть зручний:

### 5.1. Браузерний (`client_usb.html`)

Відкрийте `client_usb.html` у **Chrome/Edge** (потрібен Web Serial API),
натисніть «Підключити», оберіть COM-порт ESP32.

> Якщо Web Serial недоступний з `file://` — запустіть `python -m http.server`
> у теці й відкрийте `http://localhost:8000/client_usb.html`.

### 5.2. Нативний `.exe` (GUI, без браузера) — рекомендовано

Тека **`usb_client/`**, файл **`moto_gui.py`** — самостійне вікно на Tkinter
(входить у стандартний Python), відкриває COM-порт напряму (`pyserial`), **не
потребує браузера/Web Serial**. Вкладки Огляд / Дані / Прошивка / **Редактор
(HxD-стиль: Offset │ HEX │ ASCII)** / Журнал.

```bat
cd usb_client
build.bat            REM -> dist\moto_usb.exe (з іконкою, без консолі)
```

Зібраний `moto_usb.exe` **не потребує** ні Python, ні браузера у користувача.
Запуск із коду: `python moto_gui.py`. Іконка — `icon.ico` (генерується
`make_icon.py`, потрібен Pillow).

### 5.3. Веб-міст (`moto_bridge.py`)

Показує той самий `client_usb.html` у **будь-якому** браузері, а COM-порт
обробляє Python (Web Serial не потрібен). Збірка — `build_bridge.bat`.

Деталі обох `.exe` — `usb_client/README.md`.

### Протокол (`serial_api.h`, 115200 бод, відповідь одним рядком `#R#{json}`)

| Команда | Дія |
|---|---|
| `PING` / `READ` / `INFO` | ідентифікація / зчитати / усі дані |
| `AUTH <пароль>` | *(опційно)* звірити пароль `ADMIN_PASSWORD`, підсвітити статус |
| `GET33` / `GET38` | сирий дамп (hex) |
| `WRITE33` / `WRITEFIX33` / `WRITE38` | запис дампа (raw / +автосуми / DS2438) |
| `RESET` / `REPAIR` / `CLEAN` / `WIPE33` | скидання / ремонт / очистка / стирання |
| `SETCAP <%>` / `SETMAH <мА·год>` | знос % / залишкова ємність |
| `SETMODEL <NAME>` | ручний запис моделі (3..9 символів `A–Z 0–9`) |
| `SETETM <сек>` | ETM (наробіток) → «дата першого користування» у рації |
| `TEMPLATES` | список вшитих моделей для ініціалізації (без пароля) |
| `INITBAT <MODEL> <мАг>` | ініціалізувати порожній чип як новий АКБ моделі |
| `REBOOT` | перезавантаження ESP32 |

> 🔑 **По USB пароль ОПЦІЙНИЙ** — фізичний доступ до кабелю вже є дозволом, тож
> запис працює й без нього. `AUTH <пароль>` лише звіряє пароль і підсвічує
> статус. У **веб-API** (Wi-Fi, по мережі) пароль **обов'язковий**.
>
> 📦 Великі команди запису (`WRITE33` ~1 КБ) клієнт надсилає **частинами по
> ~200 Б** із мікропаузами, тож типового UART-буфера ESP32 достатньо і жодного
> втручання в ініціалізацію прошивки не потрібно.

## 6. Керування на пристрої

Сторінки (гортати кнопкою «Вперед»): 1) головна, 2) модель/серійний,
3) дані DS2438, 4) стан/цикли, 5) дамп DS2438, 6) дамп DS2433, 7) **Дії**.

- **«Вперед» коротко** — наступна сторінка; **довго (0.8 с)** — перечитати АКБ.
- **«Назад» коротко** — попередня сторінка; на сторінці **«Дії»** — вибір
  операції: Скидання → Ремонт → Очистка → Стерти DS2433 → **Перезавантаження** →
  **Новий АКБ** для кожної вшитої моделі.
- **«Назад» довго (0.8 с)** на «Діях» — **виконати** обрану операцію
  (свідоме підтвердження).

## 7. Індикація світлодіодами

| Стан | Сигнал |
|---|---|
| Очікування | короткий зелений пульс раз на 3 с |

| Фатальний збій старту | постійний червоний |

## 8. Операції з АКБ

> Незворотні операції — під паролем (у вебі). **Спочатку резервна копія.**

**Обслуговування (безпечно для ідентичності):**
- **♻️ Скидання** — обнуляє лічильники заряд/розряд і знос (→100 %),
  калібрування зберігається; далі — рекалібрування на IMPRES-ЗП.
- **🛠 Ремонт цілісності** — перерахунок контрольних сум (заголовок Σ≡0x41,
  записи Σ≡0x5A) + відновлення дзеркала з уцілілого DS2438.
- **🔋 Заряд у мА·год** — регістр ICA DS2438. **🩺 Знос %** — запис 0x17.
- **🪪 Запис моделі** — ручний запис part number у DS2433. Модель зберігається
  у **двох** місцях: запис `0x0B` (9 байт, Σ≡0x5A) **і** ASCII-заголовок `0x23`
  (формати 4488/4493, Σ≡0x41) — оновлюються **обидві копії**, інакше рація читає
  стару. Пишуться **лише зачеплені 32-байтові сторінки**, а не весь чіп. Перевірка
  довжини **3..9** і символів **`A–Z 0–9`**. Змінює модель лише там, де вже **є**
  запис; для порожнього чипа — «Новий акумулятор» нижче. Серійний номер
  (лазерний ROM DS2438) — **read-only**.

- **🆕 Новий акумулятор** — ініціалізація **порожнього/стертого/невідомого** чипа
  «як новий». Обираєте модель зі **списку вшитих еталонів** (`templates.h`), вводите
  **ємність у мА·год** — прошивка пише повний genuine-еталон (DS2433 **і** DS2438:
  ідентичність, калібрування, автентифікація, дзеркало), обнуляє історію/лічильники,
  ставить здоров'я 100%, а вказану ємність — у ICA (поточний заряд). Вшиті шаблони:
  **PMNN4488A**, **PMNN4493A**, **PMNN4409A**, **PMNN4409B**, а також **R7**:
  **PMNN4809A**, **APLI4810C** (формат 2021; у APLI4810C дамп DS2438 знято «порожнім»,
  тож після запису калібруйте на R7-ЗП). Додати модель — вставити
  genuine-дамп у `templates.h`). Ту саму дію продубльовано в **меню дисплея**
  («Дії» → «Новий <модель>») з паспортною ємністю за замовчуванням.
  Серійник не потрібен: прошивка прив'язана до моделі. ⚠️ ADC-калібрування DS2438
  береться від донора тієї ж моделі — тонке підстроювання робіть на IMPRES-ЗП.

**Небезпечна зона (незворотні):**
- **🧹 Очистка** — стирає ВСІ дані використання/історії/статистики,
  зберігаючи модель, серійний/ID, криву калібрування, автентифікацію, дзеркало.
- **📤 Відновлення DS2433** — запис завідомо справного дампа тієї ж моделі
  (оживляє стерту АКБ).
- **🔥 Повне стирання DS2433** — весь чіп у `0xFF` (модель/ID/калібрування теж).
  АКБ не працює до запису еталона. DS2438 не чіпається.
- **✏️ Редактор байтів** — правка будь-яких байт з автовиправленням сум.
  У вебі та в нативному GUI — **HxD-стиль**: колонка зсувів, HEX (16 байт, розбито
  по 8) і синхронна **ASCII-колонка**, зручно шукати ASCII-моделі та TLV-записи.

> 🛡 **Перевірка справжності — модельно-залежна.** Блок автентифікації
> `MOTOROLA` і калібрувальний підпис `0x1B-0x1E` є лише у формату 2017
> (4488A/4493A). У **4409A** (формат 2014) їх штатно немає — тому їх відсутність
> **не** вважається підробкою. Червоні прапори для будь-якої моделі: побитий
> заголовок (Σ≠0x41), відсутній запис моделі, переповнений CCA (0xFFFF).

## 9. Одиниці вимірювання

- **Лічильники заряду — у мА·год:** `мА·год = регістр × 0.4882 / Rsense`.
  `settings.h → DS2438_RSENSE_OHM` (за замовч. 0.025 Ом) — підберіть під АКБ.
- **Цикли:** 1 цикл = сумарний заряд (розряд) = одна повна ємність.
  `settings.h → BATTERY_RATED_MAH` (2500; PMNN4409 — 2500, PMNN4493 — 3000,
  R7/PMNN4808A — ~2450).
- **Заряд %:** пріоритет ICA (DS2438), інакше — оцінка за напругою.
- **Дата першого користування** НЕ зберігається в АКБ — рація рахує її як
  `поточний час рації − ETM`, де **ETM** = лічильник наробітку DS2438 (`[8..11]`,
  сек). Перевірено діффом однієї АКБ до/після калібрування: дата не змінилась, а
  ETM виріс рівно на тривалість циклу (~7 год). Тому дата **не впливає** на
  «оригінальність» (це похідне поле), а змінюється записом ETM = `сьогодні −
  бажана дата` (клієнт рахує це сам; поле «📅 Дата» у веб/USB/GUI). ESP32 без RTC
  показує лише **наробіток** (вік), а календарну дату обчислює клієнт.

## 10. Формат прошивки IMPRES

Розбір дампів (PMNN4488A / 4493A / 4409A / 4409B, PT-клон, стертий) встановив:

- **Заголовок DS2433:** `сума байт 0x00..0x1F ≡ 0x41` (байт 0x1F — контрольний).
- **TLV-записи:** сума байт запису разом з контрольним байтом `≡ 0x5A`
  (довжина запису = значення байта-тега: `0x0B`=11, `0x0D`=13, `0x16`=22, `0x17`=23).
- **Дзеркало калібрування:** `DS2438[24:50] == DS2433[1:27]` — однакове для
  всіх батарей однієї моделі (4488A і 4493A збігаються; 4409A інше).
- **Модель:** запис `0x0B` (`0x0B` + літера) у другій половині + ASCII-заголовок
  `0x23` (лише формати 4488/4493). CCA/DCA дзеркаляться у записі `0x0D`.

**Прошивка прив'язана до МОДЕЛІ, а не до серійного номера чипа** — перевірено
на дампах 3 незалежними тестами:
1. **Калібрувальний блок однаковий при різних серійниках.** Два PMNN4493A з
   різними ROM-серійниками (`A6964F…` / `A6CCB5…`) мають **побайтово ідентичний**
   блок `DS2433[0x01:0x1A]`. Прив'язки до серійника тут немає.
2. **Байтів серійника немає в пам'яті.** Унікальні 6 байт ROM (без family `A6`
   і CRC) **не зустрічаються** ні в DS2433, ні в DS2438 — ні прямо, ні реверсом.
3. **Емпірично:** повне стирання DS2433 + «Ремонт» — рація все одно опізнає
   модель (ROM DS2438 не чіпається). Отже перехресної перевірки «підпис DS2433 ↔
   серійник DS2438» немає.

Тому відновлення можливе з еталона тієї ж моделі, без підбору серійника.

> ⚠️ Автопідбір/«розлочка» без еталона неможливі: калібрувальні дані унікальні
> для моделі й не виводяться математично — потрібен заведомо робочий донор
> тієї ж моделі.

### Блокування після заміни елементів (IMPRES «обучення»)

IMPRES — **обучаєма** система. Крім моделі-калібрування, у DS2433 є **learned-
дані**: лічильники `0x0D` (CCA/DCA), крива `0x17`, телеметрія в кінці (запис
`0x0B` — виміряні за життя імпеданси/просадки; у двох 4493A вони починаються
однаково `6D 80 … 84 BD … 72` і різняться лише у виміряних байтах, тобто це НЕ
хеш серійника). Рація в реальному часі звіряє струм/напругу елементів із цими
learned-даними + кривою розряду.

Після **заміни елементів** нові банки мають іншу ємність/імпеданс, DS2438
рахує «поверх» старих даних → рассогласування → рація помічає пакет як
**«невідомий»** і вимикає гейдж. Це захист IMPRES, а не поломка прошивки.

- **Стирання** → рація не бачить IMPRES-даних, робить грубий вольтметр-гейдж;
  ЗП не опізнає пакет → калібрування не запускає.
- **Стирання + Ремонт** → структура валідна, модель опізнається, гейдж працює,
  рація просить «відновити» (стан **прийнято, але не відкалібровано**). Це
  найближчий досяжний прошивкою стан.
- **«Новий АКБ» з донора** → у чіп потрапляють learned-дані **донора**; на нових
  банках вони не сходяться → миттєвий reject («невідомий»). Гірше за чистий
  стан.

> 🔑 **Обійти фізичну калібрування неможливо:** зарядний пристрій IMPRES має сам
> провести цикл (повний розряд-заряд), виміряти **реальну** ємність нових банок
> і перезаписати learned-дані. Прошивка може лише привести пакет у стан
> «валідний, не відкалібрований», далі — калібрування на IMPRES-ЗП.

**Що саме пише ЗП (перевірено діффом однієї АКБ ДО/ПІСЛЯ калібрування):**
у стерту (0xFF) зону DS2433 з'являється запис **`0A` (learned-калібрування,
Σ≡0x5A)** — виміряні ємність/імпеданс РЕАЛЬНИХ банок — плюс запис статусу `0D`;
оновлюються кінець кривої розряду, лічильники `0x0D` CCA/DCA і кінцева
телеметрія. Тобто «learned» дані ЗП **вимірює** — підробити під нові банки не
можна. Але щоб пакет ПРИЙНЯВСЯ і ЗП ВЗЯВ його на калібрування, треба **стерти
старий запис `0A`** (він суперечить новим банкам) і обнулити лічильники — це й
робить «🔧 Ремонт після заміни елементів». Саме тому «Новий АКБ» раніше давав
«невідома»: лишав learned-`0A` донора; тепер він теж стирає його автоматично.

## 11. Типові сценарії відновлення

- **АКБ «невідома», але чіп читається** → «🛠 Ремонт цілісності». Якщо суми і
  дзеркало в нормі — проблема не в прошивці, а в елементах або треба
  калібрування на IMPRES-ЗП.
- **Порожній / стертий / зовсім новий чіп** → «🆕 Новий акумулятор»: оберіть
  модель + впишіть ємність — запишеться повний робочий еталон (DS2433+DS2438).
- **DS2433 пошкоджений/стертий** → «📤 Відновлення DS2433» (еталон тієї ж
  моделі) + «🛠 Ремонт» (синхронізація дзеркала з DS2438).
- **Чіп остаточно «зіпсований»** → «🔥 Повне стирання DS2433» → запис еталона.
- **Прибрати історію/знос, лишити робочою** → «🧹 Очистка».
- **Після заміни елементів** (рація пише «невідома») → **«🔧 Ремонт після заміни
  елементів»** (веб/USB/меню «Дії» → «Рекалібр.»). Стирає старе learned-
  калібрування (запис `0x0A`) і обнуляє лічильники, лишаючи модель/криву/ID →
  пакет стає «валідний, не відкалібрований». Потім — **калібрування на IMPRES-ЗП**
  (ЗП виміряє нову ємність). Обійти фізичну калібровку прошивкою неможливо
  (див. розділ 10). *(Раніше «Новий АКБ» давав «невідома», бо лишав learned-дані
  донора — тепер `Новий АКБ` теж стирає їх автоматично.)*
